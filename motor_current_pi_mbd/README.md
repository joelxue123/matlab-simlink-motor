# Motor Current PI MBD

This directory contains the first reusable current-loop PI controller module
for the motor MBD workflow.

It is intentionally implemented with Simulink primitive blocks instead of a
MATLAB Function block or a black-box PID Controller block.

## Architecture

```text
current_pi_input_t
  -> CurrentPiStep
  -> current_pi_output_t
```

Input bus:

```text
current_pi_input_t {
  id_ref
  iq_ref
  id_meas
  iq_meas
  omega_e
  vdc
}
```

Output bus:

```text
current_pi_output_t {
  vd_ref
  vq_ref
}
```

The current milestone implements:

```text
error = ref - meas
u_pre_sat = Kp * error + integrator
u_sat = clamp(u_pre_sat, -0.577 * vdc, 0.577 * vdc)
integrator_next = integrator + Ts * (Ki * error + Kaw * (u_sat - u_pre_sat))
```

## Files

- `build_current_pi_model.m` creates `current_pi_interface.sldd` and
  `current_pi_model.slx`.
- `run_current_pi_smoke_test.m` builds and simulates the model, then checks
  output range and finite values.
- `current_pi_interface.sldd` is generated by the build script.

## Saturation / Anti-Windup Test

Run:

```bash
matlab -batch "run('motor_current_pi_mbd/run_current_pi_saturation_test.m')"
```

Plot PI output data:

```bash
matlab -batch "run('motor_current_pi_mbd/plot_current_pi_saturation_outputs.m')"
```

The plot is saved to:

```text
motor_current_pi_mbd/reports/current_pi_saturation_outputs.png
```

Analyze saturated-output ripple:

```bash
matlab -batch "run('motor_current_pi_mbd/analyze_current_pi_saturation_ripple.m')"
```

Current ripple result:

```text
With anti-windup:
  saturation vq_ref p-p/std = 0 / 0 V
  settled saturation vq p-p/std = 0 / 0 V

Without anti-windup:
  saturation vq_ref p-p/std = 0 / 0 V
  settled saturation vq p-p/std = 0 / 0 V
```

Conclusion: saturated `vq_ref` itself is not creating high-frequency output
ripple in this harness. Anti-windup mainly changes the post-release recovery
behavior.

Plot near-saturation output behavior:

```bash
matlab -batch "run('motor_current_pi_mbd/plot_current_pi_near_saturation_outputs.m')"
```

The plot is saved to:

```text
motor_current_pi_mbd/reports/current_pi_near_saturation_outputs.png
```

Current near-saturation result:

```text
v_limit                  = 6.92400026 V
iq_ref                   = 7 A
iq_ref - limit equivalent = 0.0759997368 A (1.09763%)

With anti-windup:
  vq_ref p-p/std = 0 / 0 V

Without anti-windup:
  vq_ref p-p/std = 0 / 0 V
```

This test connects `CurrentPiStep` to a simple q-axis first-order current plant.
It drives `iq_ref` high enough to saturate the voltage output, then releases the
reference back to zero. The script compares default `Kaw_iq` against
`Kaw_iq = 0`.

Current result:

```text
With anti-windup:
  final iq                 = 6.05617e-07 A
  release saturation count = 39
  release recovery time    = 0.00675 s
  release |vq| area        = 0.021076 V*s

Without anti-windup:
  final iq                 = 6.924 A
  release saturation count = 561
  release recovery time    = Inf s
  release |vq| area        = 0.193872 V*s

Current PI saturation anti-windup test passed.
```

Conclusion:

```text
Back-calculation anti-windup prevents the q-axis integrator from staying wound
up after the voltage command saturates. With Kaw_iq = 0, the controller remains
stuck near the positive voltage/current limit after the reference is released.
```

## Design Notes

- Parameters `Kp_id`, `Ki_id`, `Kaw_id`, `Kp_iq`, `Ki_iq`, `Kaw_iq`, and
  `VLimitRatio` live in the `.sldd`.
- Controller-facing scalar types are `Simulink.AliasType('single')`.
- The sample time remains in `current_pi_simcfg.Ts_ctrl` as a double-valued
  simulation scheduler parameter.
- `omega_e` is part of the input bus for future decoupling feedforward. It is
  not used in this first PI-only milestone.
- The next integration step should add `vd/vq` decoupling and a proper dq
  vector voltage limiter before connecting this to `DqToAbcDutyStep`.
